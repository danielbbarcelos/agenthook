Terminal interface · design spec

agenthook — TUI redesign

A reference design for the agenthook command-line interface: ASCII screen mockups, the navigation map, the ANSI palette, microcopy, and terminal interaction patterns — ready to implement in Typer · Rich · questionary, with the live screens flagged for a future Textual dashboard.

Target responsive 120 → 80 cols Box rounded ╭─╮ Palette amber / earth Microcopy English Pointer ●
What this redesign fixes. Five pains drove it: getting lost in deep menus, no live view of a running job, an intimidating instance/auth setup, weak state feedback, and a scrollback that turns to mush. The answers, throughout: a persistent breadcrumb, a smart screen-clear rule, a guided setup wizard with a progress rail, a shared status vocabulary, and a stepped job runner.
01

Design system

Everything is monospace and color-coded by role, never by decoration. The palette is warm — amber on near-black earth — so success-green and response-cyan read as deliberate accents rather than noise. Color always pairs with a glyph or word, so meaning survives on a mono/no-color terminal.

ANSI palette — roles
highlight / brand
#d9a441 · amber
Selection, pointer ●, focused field, brand. The one color that means “you are here / act here.”
ok / success
#a3be8c · sage
Completed jobs, logged-in auth, confirmations, passing checks.
info / response
#88c0d0 · cyan
Agent replies in chat, hints, neutral values, links.
warning / awaiting
#d08770 · clay
awaiting_approval, blocked, failed-checks — needs a human, not an error.
error / failed
#bf616a · rust
error, timeout, destructive confirmations, missing auth.
plan / session
#b48ead · lilac
Plans awaiting review and durable sessions/threads — a distinct “thinking” hue.
muted
#6f6a5d · stone
Back ←, quit, secondary labels, separators, nav hints, queued.
foreground
#e8e3d8 · bone
Default body text on #15140f background.

02

Interaction patterns

Two levels, always. Every screen here is reachable two ways: a dry command for automation/CI, and a friendly path through arrow-key menus. Running agenthook with no args opens the main menu; any command with a required argument omitted opens a picker for it.

Two-level principle
# dry — scriptable, no prompts $ agenthook instances view api-bot $ agenthook jobs approve 3f1a --yes # friendly — omit the arg, get a picker $ agenthook instances view ? Which instance? (type to filter) ● api-bot claude-code · pr docs-writer aider · analysis infra-bot codex · commit # bare — the full menu $ agenthook
Keys — consistent across every menu
↑ ↓ / j k move cursor ↵ enter select esc back one level ← back (visible menu item) / or type filter the current list q quit from main menu ctrl-c cancel current prompt tab accept suggested default in chat / shell: /exit leave the container /help commands · ↑ history
Screen-clear & history rule

The old build never cleared, so the scrollback became soup. New rule: clear on “big” transitions, accumulate on “small” ones — and always anchor with a breadcrumb so the user knows where they are without re-reading history.

clear + redraw entering a new top section (instances / jobs / sessions), opening view/edit, launching chat or shell. keep + append steps inside one flow (wizard pages, a sequence of prompts, confirmations) — so context stays visible. breadcrumb first line after every clear, dimmed: agenthook ▸ instances ▸ api-bot ▸ edit

04

Entry banner & main menu

Drawn once on launch. The node mark reads as a webhook → agent chain. The status line answers the first question anyone has — is anything running, is the server up — before they touch a key. serve is promoted here so the core action (start receiving webhooks) is one keystroke from the top.

$ agenthookno args → menu
●─╮
  ╰─●─╮        agenthook  v0.4.2
      ╰─●      self-hosted agent task runner

3 instances · 1 running · 2 queued · server ● up :8080

  ? Where to?

● instances   register, configure & run agents
  jobs        watch and review executions
  sessions    durable threads across jobs
  serve       start the webhook server
  ───  ──────────────────────────
  quit

Interaction

  • ↑↓ move, ↵ select, q quits from here only.
  • Screen clears once on entry; the banner is not redrawn as you navigate.
  • Each item is also a subcommand — agenthook jobs jumps straight in.

Microcopy & hierarchy

  • Prompt “Where to?” — friendly, not a label.
  • Status counts use the shared color vocabulary: running, queued, ● up.
  • Descriptions are verb-first and ≤6 words.
05

Instances submenu

The fix for “too many levels”: instead of a flat 8-item list, actions are grouped by intent — what you do with an agent (interact), how you set it up (manage), and bulk list/remove. Labelled separators give the eye anchors; back is always the muted last row.

$ agenthook instancesgrouped
agenthook ▸ instances

  ? instances — pick an action

  ─── interact ──────────────────────
● chat        talk to the agent in a container
  shell       open a bash shell in a container
  ─── manage ────────────────────────
  add         register a new instance
  view        inspect config, repos, env & jobs
  edit        change config or authentication
  ─── list · remove ─────────────────
  list        all instances at a glance
  remove      delete an instance
  ───  ──────────────────────────
  ← back

Interaction

  • Groups are visual only — arrows still move through every item linearly.
  • esc and the ← back row both return to the main menu.
  • Typing filters across all groups at once.

Microcopy & hierarchy

  • Separators are titled: ─── interact ───.
  • Back uses ← + muted, never a bracketed [Back].
  • Action names mirror the subcommands exactly.

Add instance — guided setup

The “intimidating setup” pain, solved with a progress rail: one decision per page, with completed steps collapsing into their chosen values on the left so you always see how far you’ve come and what you picked. Auth isolation is stated inline at the moment it matters. The encryption key gets its own emphatic, framed step — shown exactly once.

Interaction

  • Each step is its own prompt; esc walks back without losing earlier answers.
  • Sensible defaults pre-selected — ↵ through the happy path in seconds.
  • The key step refuses to advance until you confirm you saved it.

Microcopy & hierarchy

  • Rail glyphs: ✓ done · ▸ current · · pending.
  • Reassurance copy: “never the host’s.”
  • Warning header ▲ shown once sets expectation before the secret appears.

View instance

Everything about one instance on a single screen, sectioned by titled rules so density stays scannable. Secrets are always obfuscated (••••) and tagged secret. The recent-jobs strip turns this into a launchpad — IDs are ready to paste into jobs view.

$ agenthook instances view api-botread-only
agenthook ▸ instances ▸ api-bot

  ● api-bot   active   ·  last run 14m ago

  ─── config ────────────────────────────────────────────────
  engine        claude-code
  auth          subscription  

  deliverable   pr
  branch base   main
  model         default

  ─── repositories  (2) ─────────────────────────────────────
  • acme/web-app        main
  • acme/design-sys     main

  ─── env  (3, encrypted) ───────────────────────────────────
  GITHUB_TOKEN   ••••••••••••   secret
  NPM_TOKEN      ••••••••••••   secret

  ─── recent jobs ───────────────────────────────────────────
  3f1a  success        pr       2h ago
  9c20  running        patch    now

Interaction

  • Read-only; e jumps to edit, esc back to the instances menu.
  • Job IDs are copy-paste targets for jobs view <id>.
  • Long repo / env lists scroll within the section, header pinned.

Microcopy & hierarchy

  • Header line carries live status: ● active + relative “last run”.
  • Counts in section titles: repositories (2).
  • Secrets never shown — value column reads •••••••• + secret.
08

Edit instance

A field picker, not a wall of form inputs. Each row shows its current value on the right, so you edit by recognition. Three rows are doors into dedicated editors (auth, repos, env). pause / resume is a single state toggle that relabels itself.

$ agenthook instances edit api-botfield picker
agenthook ▸ instances ▸ api-bot ▸ edit

  ? What do you want to change?

  ─── config ──────────────────────
● deliverable   pr
  engine        claude-code
  model         default
  branch base   main
  ─── connections ─────────────────
  authenticationlogged in
  repositories  2 repos
  env vars      3 vars
  ─── state ───────────────────────
  pause         running — pause intake
  ───  ──────────────────────────
  ← done

Interaction

  • Pick a field → focused prompt for just that value → returns here.
  • authentication / repositories / env vars open their own screens.
  • ← done writes nothing extra; every edit is saved on confirm.

Microcopy & hierarchy

  • Current value shown muted beside each field — edit by recognition.
  • Toggle copy flips with state: pause ⇄ resume.
  • “done” not “back” — signals the edit session is complete.
09

Instance authentication

Auth isolation made visible: the panel states scope: this instance only so it’s obvious nothing is inherited from the host. The action set covers the full lifecycle — switch method, re-login, or log out (which clears credentials and lets a fresh login start clean).

$ agenthook instances edit api-bot · authenticationisolated
agenthook ▸ instances ▸ api-bot ▸ authentication

╭─ Authentication · api-bot ─────────────────────────────╮
│  status   ● logged in                                  │
│  method   subscription (isolated)                      │
│  account  agent-bot@acme.dev                           │
│  scope    this instance only                           │
╰────────────────────────────────────────────────────────╯

  ? Manage authentication

● switch      use an API key instead
  relogin     re-run the isolated login
  logout      clear credentials & data
  ───  ──────────────────────────
  ← back

Interaction

  • switch reveals an api-key prompt; logout asks to confirm.
  • Login opens an isolated device-code / browser flow — never the host session.
  • State refreshes in place after any action.

Microcopy & hierarchy

  • Status uses the auth vocabulary: logged in · via api-key · not logged in.
  • Explicit (isolated) and scope: this instance only.
  • logout copy warns it clears credentials & data.
10

Env vars editor

Secrets are encrypted at rest and shown obfuscated; the secret flag is set per-variable at write time. The set flow is three quick prompts — name, value, secret? — so adding a token is a 10-second job.

$ agenthook instances edit api-bot · envobfuscated
agenthook ▸ instances ▸ api-bot ▸ env vars

  ─── environment  ·  3 variables ───────────────────
  GITHUB_TOKEN   ••••••••••••   secret
  NPM_TOKEN      ••••••••••••   secret
  LOG_LEVEL      info

  ? Manage variables

● set       add or update a variable
  remove    delete a variable
  ───  ──────────────────────────
  ← back

  — on “set” —
  ? Name   API_BASE_URL
  ? Value  https://api.acme.dev
  ? Secret? (encrypt & obfuscate)  [Y/n]

Interaction

  • set on an existing name updates it; new names are created.
  • Secret values never echo to the terminal as you type.
  • remove confirms by name before deleting.

Microcopy & hierarchy

  • Non-secret values show in plain; secrets as •••• + secret.
  • Default for Secret? is Y — safe by default.
  • Listing always shows the count: 3 variables.

Repo pool editor

An instance can target 0..N repositories. The pool is a flat list with default branches; add/remove are the only verbs. A job picks its repo from this pool, so the pool is the instance’s blast radius.

$ agenthook instances edit api-bot · repos0..N
agenthook ▸ instances ▸ api-bot ▸ repositories

  ─── repository pool  ·  2 ───────────────────────
  • acme/web-app       main
  • acme/design-sys    main

  ? Manage repositories

● add     grant access to a repo
  remove  revoke a repo
  ───  ──────────────────────────
  ← back

  — on “add” —
  ? Repository (owner/name)  acme/api-gateway▏
  ? Default branch main

Interaction

  • add validates the repo against the instance’s auth before saving.
  • remove lists current repos as a picker.
  • An empty pool is allowed — analysis jobs can run without one.

Microcopy & hierarchy

  • Bullet • for each repo; branch shown muted.
  • Prompt hints the format: (owner/name).
  • Count in the section title: repository pool · 2.
12

Jobs — live list

The headline “can’t see what’s happening” fix at the list level. --watch refreshes in place every 2s; the full status vocabulary is color-coded so the eye finds the awaiting and error rows instantly. This is a prime Textual candidate — a real-time table with row actions.

$ agenthook jobs --watchlive · Textual
agenthook ▸ jobs

  jobs   ● live   ↻ refreshing every 2s

  ID    INSTANCE      STATUS             DELIVERABLE   AGE
  ────────────────────────────────────────────────────────────────
  9c20  api-bot      ▸ running         patch        0:42
  a18e  api-bot      ◷ awaiting_approvalpr           3m
  3f1a  api-bot      ✓ success         pr           2h
  8b07  docs-writer  ▲ failed-checks   pr           1d
  77d1  infra-bot    ✗ error           commit       1d
  5e0c  api-bot      · queued          analysis     —

  ↵ view   a approve   c cancel   / filter   esc back

Interaction

  • ↵ opens detail, a approves an awaiting job, c cancels.
  • / filters by instance, status, or deliverable.
  • Without --watch it prints a static snapshot for CI/pipes.

Microcopy & hierarchy

  • Status glyphs double the color: ▸ running, ◷ awaiting, ✓, ▲, ✗.
  • Running jobs show a live elapsed clock (0:42).
  • Header row + rule; no vertical borders keeps it light at 80 cols.

Job runner & plan approval

The core state-feedback fix. A running job shows a stepped rail (clone → plan → edit → checks → deliver) with a compact stats line (elapsed · cost · tokens) and a tail of the agent’s actions — quiet by default, full stream on --follow. When the engine pauses, the job enters awaiting_approval and the plan is shown inline with approve / edit / reject.

$ agenthook jobs view 9c20live · Textual
agenthook ▸ jobs ▸ 9c20

  9c20   api-bot   ·  patch   ·  acme/web-app

  ✓ clone ── ✓ plan ── ▸ edit ── · checks ── · deliver

  ◷ running   elapsed 0:42   ·  $0.018   ·  12.4k tok

  ─── agent log ─────────────────────────────────────────────
  ◆ reading src/build/nightly.ts
  ◆ found race: cache write not awaited
  ◆ editing 2 files…
  › src/build/nightly.ts   +6 −2

  follow live: agenthook jobs logs 9c20 --follow
  ↵ detail   c cancel   esc back
When the engine pauses for review

agenthook ▸ jobs ▸ a18e ▸ approve

  ◷ awaiting_approval   the agent paused for review

╭─ Plan · a18e ────────────────────────────────────────────╮
│  1. Add retry wrapper around cache.flush()               │

  ? Approve this plan?

● approve   run the plan as proposed
  edit      send a note back & replan
  reject    stop the job here

  ↑↓ move   ↵ select

Interaction

  • Step rail: ✓ done · ▸ running · · pending.
  • --follow streams every action; bare view shows the last few.
  • Approval is inline — edit lets you send guidance and replan.

Microcopy & hierarchy

  • Stats line is always one row: elapsed · cost · tokens.
  • Diff hunks colored +6 −2; file paths in fg.
  • Plan panel uses the lilac “thinking” hue, distinct from running amber.

Sessions — threads across jobs

A session is a durable thread keyed by something stable (a ticket like PROJ-481), so context carries across many jobs over time. The list answers “which threads are alive”; the view shows the thread’s brief plus its full job history, and lets you resume in chat right where it left off.

$ agenthook sessionsindex
agenthook ▸ sessions

  sessions   durable threads shared across jobs

  THREAD KEY      INSTANCE      JOBS   LAST
  ──────────────────────────────────────────────────
  PROJ-481       api-bot      4     2h ago
  PROJ-455       api-bot      2     yesterday
  INFRA-12       infra-bot    7     3d ago

  ↵ view thread   / filter   esc back
Opening one thread
$ agenthook sessions view PROJ-481thread
agenthook ▸ sessions ▸ PROJ-481

  PROJ-481   api-bot   ·  4 jobs   ·  opened 6d ago

  ─── thread context ────────────────────────────────────
  “Stabilise the nightly build pipeline.”

  ─── jobs in this thread ───────────────────────────────
  3f1a  ✓ success        analysis  6d ago
  6a22  ✓ success        patch     5d ago
  8b07  ▲ failed-checks  pr        1d ago
  9c20  ▸ running        patch     now

  ↵ open job   r resume in chat   esc back

Interaction

  • ↵ opens a thread; inside, ↵ opens a job, r resumes in chat.
  • / filters by key or instance.
  • Thread keys map to external IDs (tickets, PR numbers).

Microcopy & hierarchy

  • Thread key carries the session lilac throughout.
  • Job count + relative “last” give freshness at a glance.
  • Context line quotes the thread’s original ask.

Interact — chat REPL & shell

“Entering an instance” spins up an ephemeral, isolated Docker container per the security model. chat is a multi-turn REPL with the agent (optionally bound to a session thread); shell drops you into bash in the same kind of container. Both are strong Textual candidates for a richer live pane.

$ agenthook chat api-bot PROJ-481REPL · Textual
agenthook ▸ chat ▸ api-bot ▸ #PROJ-481

  ◇ container ready   ·  acme/web-app @ main   ·  /exit to leave

  you  why is the nightly build flaky?

  ◆ api-bot  It’s a race in the cache flush — the write
             isn’t awaited, so the build sometimes reads
             stale artifacts. Want me to open a patch?

  you  yes, and add a regression test

  › ▏

  /exit leave   /help commands   ↑ history
Or drop straight into a shell
$ agenthook shell api-botcontainer
agenthook ▸ shell ▸ api-bot

  ◇ ephemeral container   ·  acme/web-app @ main   ·  /exit to leave

  agent@api-bot:/workspace$ git status
  On branch agent/nightly-fix
  Changes not staged for commit:
    modified:   src/build/nightly.ts

  agent@api-bot:/workspace$ ▏

  type /exit to leave  ·  changes here do not persist unless committed

Interaction

  • Multi-turn; ↑ recalls input history, /help lists slash-commands.
  • /exit tears down the container and returns to the menu.
  • Passing a thread key binds the chat to a durable session.

Microcopy & hierarchy

  • Agent replies in cyan with a ◆ marker; your turns are muted-labelled.
  • Banner states isolation: ephemeral container.
  • Shell warns: changes don’t persist unless committed.
